# 智能学习预警服务系统 — 功能模块详细介绍

## 一、系统总体架构

本系统基于 Spring Boot + Spring Cloud Alibaba 微服务架构开发，共包含 13 个微服务模块，通过 Nacos 进行服务注册与发现，由 Spring Cloud Gateway 统一接入。系统覆盖学生、教师（班主任）、辅导员、管理员、学院管理员五类角色，构成"成绩管理→智能预警→帮扶计划→数据分析"的完整业务闭环。前端覆盖 Web 管理端（Vue 3 + Element Plus）和微信小程序端（UniApp Vue 3），后端服务通过 OpenFeign 实现跨服务协同调用。

## 二、基础支撑模块

### 2.1 认证与鉴权模块

**核心职责：** 负责用户身份认证、JWT Token 签发与续期、角色权限管控。

**操作逻辑：**
- 用户通过登录页提交学号/工号及密码，系统校验凭证后生成 JWT Token 返回客户端。
- Token 附带角色编码（学生=1、教师=2、辅导员=3、管理员=4、学院管理员=5），前端和网关根据角色路由前缀（`/student/*`、`/teacher/*` 等）实现权限隔离。
- 注册流程按角色分流：学生注册需选择学院、专业、班级三级联动；教师注册需选择学院和职称；辅导员注册需选择学院。
- Gateway 层对所有请求进行 JWT 验签，拦截未授权访问。

**模块间交互：**
- `user-service`（8082）负责登录注册和 JWT 签发，将用户基础信息写入 `users` 表。
- `gateway-service`（8076）作为流量入口，对 `/api/auth/**` 路径放行，其余路径统一校验 Token。
- `auth-service`（8072）管理角色与权限的映射关系，供管理员在角色管理页面进行权限分配。

### 2.2 基础数据管理模块

**核心职责：** 管理系统基础架构数据，包括学院、专业、班级、课程的增删改查。

**操作逻辑：**
- 管理员在 Web 端通过学院管理、专业管理、课程管理三个页面维护基础数据。
- 录入数据时支持关联校验：专业必须隶属于一个学院，班级必须关联专业和年级。
- 课程信息包括课程名称、学分、所属学院，支持按学院筛选检索。

**模块间交互：**
- `admin-service`（8070）提供 CRUD 接口，直接操作 `colleges`、`majors`、`classes`、`courses` 等基础表。
- `college-service`（8073）为其他服务提供学院和专业信息的只读查询接口。
- 学生注册和教师分配班级时，前端通过 `college-service` 获取下拉选项数据，实现学院→专业→班级三级联动选择。

## 三、学生端功能模块

学生端是系统的核心信息消费端，覆盖 Web 端和小程序端，提供学业看板、成绩查询、预警管理、帮扶计划、成绩申诉、对标分析、数据分析等功能。

### 3.1 学业看板

**核心职责：** 为每个学生提供一站式学业概况视图。

**操作逻辑：**
- 登录后首页展示蓝色风格 Hero 区域，包含姓名、角色、GPA 绩点、已修学分、预警数量等核心指标。
- 下方排列四个快捷图标入口：成绩查询、预警管理、帮扶计划、数据分析。
- "最近预警"卡片展示最新三条预警记录，每条标注等级（严重/中等/轻微），点击"查看全部"跳转到预警管理页。
- "帮扶计划"卡片展示当前进行中的帮扶计划及进度条，支持快速了解帮扶状态。

**交互关系：**
- 首页加载时调用 `student-service`（8079）的 `/students/dashboard/{userId}` 接口获取 GPA、学分、预警数、近期预警等聚合数据。
- 点击快捷入口时，TabBar 页面（成绩、预警）使用 `switchTab` 导航，非 TabBar 页面（帮扶、分析）使用 `navigateTo` 导航。

### 3.2 成绩查询

**核心职责：** 提供按学期筛选的成绩明细，展示课程名称、分数、绩点。

**操作逻辑：**
- 顶部显示可横向滚动的学期标签栏，标签名称为"2025-2026-02""2025-2026-01"等学年学期格式，当前选中学期底部有红色指示线。
- 下方为蓝色表头白色背景的成绩表格，列包括：课程名称、分数、绩点。分数 ≥90 分显示"优秀"，其余显示实际分数数值；绩点按 4.0 制 GPA 公式自动计算。
- 点击不同学期标签时，前端携带 `semester=2025-2026-2` 参数请求后端，后端解析出年份和学期后，从 `scores` 表中筛选对应学期的成绩记录。
- Web 端额外提供学期下拉框筛选、GPA 趋势折线图、成绩分布饼图、导出 Excel 功能。

**交互关系：**
- `student-service` 的 `ScoreMapper` 从 `scores` 表查询成绩，支持按 `year` 和 `semester` 双条件过滤。
- 教师端录入成绩时，通过 `teacher-service` 的同步机制同时写入 `score_record` 和 `scores` 两张表，保证学生端数据即时可见。

### 3.3 预警管理

**核心职责：** 展示学生个人的所有学业预警记录，包括等级、原因、时间等信息。

**操作逻辑：**
- 列表页每行展示一条预警，左侧为等级颜色圆点（红色=严重、橙色=中等、蓝色=轻微），中间为预警标题和描述，右侧为等级标签。
- 点击任意预警记录弹出详情窗，展示预警等级、描述、创建时间等完整信息。
- 预警状态由预警引擎自动生成，学生无权修改或删除，仅可查看。

**交互关系：**
- `warning-service`（8093）根据管理员配置的预警规则，定时扫描 `score_record` 中的数据，对触发条件的学生生成预警记录并写入 `academic_warnings` 表。
- 预警生成后，通过 RabbitMQ 异步通知 `message-service` 向学生和对应辅导员推送预警消息。

### 3.4 帮扶计划

**核心职责：** 查看辅导员为自己制定的帮扶计划，更新计划进度，提交帮扶反馈。

**操作逻辑：**
- 列表页展示所有帮扶计划，每项计划包含标题、状态标签（进行中/已完成/待开始）、进度条、起止日期。
- 进行中的计划提供"更新进度"按钮，打开滑块弹窗可拖动进度百分比并提交。
- 计划完成后可对帮扶效果进行评价反馈。

**交互关系：**
- `counselor-service`（8074）创建帮扶计划，关联学生 ID 和辅导员 ID。
- 学生通过 `student-service` 查询自己的帮扶计划并更新进度，进度数据回写到 `assistance_plans` 表。
- 帮扶评价由 `AssistanceEvaluation` 实体承载，学生提交后辅导员可在其管理端查看。

### 3.5 成绩申诉

**核心职责：** 对有异议的成绩提交申诉，查看申诉处理状态。

**操作逻辑：**
- 上半部分为申诉提交区：下拉选择课程、输入申诉理由（限 200 字），提交后进入待处理状态。
- 下半部分为申诉列表：每条申诉展示课程名称、提交日期、处理状态（待处理/已通过/已驳回）。
- 管理员审核后，申诉状态更新，学生可实时查看结果。

**交互关系：**
- 申诉请求由 `student-service` 接收并写入 `score_appeals` 表。
- `admin-service` 在申诉管理页面进行审核，审核后通过 `message-service` 通知学生。

### 3.6 对标分析

**核心职责：** 帮助学生了解自己在班级和专业的成绩排名，明确学习定位。

**操作逻辑：**
- 展示当前学期的班级排名、专业排名、班级均分对比、个人 GPA 与班级均分差距。
- Web 端以图表形式展示成绩分布和排名趋势。

**交互关系：**
- `student-service` 通过 `BenchmarkAnalysisMapper` 查询对标分析数据，调用 `ScoreMapper` 的班级排名计算 SQL 获取加权 GPA 排名。

### 3.7 个人中心

**核心职责：** 小程序底部第四个 Tab"My"整合个人信息、快捷入口和退出登录功能。

**操作逻辑：**
- 顶部蓝色渐变区域展示头像、姓名、角色。
- 中间显示 GPA、已修学分、预警数量三个核心数据卡片。
- 下方功能列表包含：帮扶计划、成绩申诉、数据分析、消息中心、个人设置、退出登录六个功能入口。
- 点击退出登录后清除本地 Token 和缓存数据，跳转到登录页。

## 四、教师端功能模块

教师端承担教学管理核心职责，提供成绩管理、课程管理、班级管理、预警处理、数据分析等功能。

### 4.1 成绩管理

**核心职责：** 教师录入、修改、导入、导出学生成绩，是学生成绩数据的唯一入口。

**操作逻辑：**
- 首先选择目标课程和班级，系统加载该班级的学生名单。
- 在表格中逐行录入或修改学生成绩，支持批量导入 Excel。
- 单条修改成绩时，系统记录修改前后的数值，写入审计日志。
- 成绩保存后，系统自动同步到 `scores` 表，确保学生端和辅导员端可实时查看。

**交互关系：**
- 成绩数据写入 `score_record` 表（原有逻辑），同时通过 `syncToScores()` 方法同步写入 `scores` 表（新增同步逻辑），两张表保持数据一致。
- 修改成绩时，`ScoreAuditLogService` 记录审计日志，包含修改人、修改前后值、修改原因。
- `warning-service` 读取 `score_record` 数据执行预警扫描，发现不及格或多科挂科等情况下生成预警。

### 4.2 班级管理

**核心职责：** 教师申请管理班级，查看班级学生名单和成绩概况。

**操作逻辑：**
- 教师在班级管理页提交管理申请，管理员或学院管理员审核通过后生效。
- 管理权限生效后，教师可查看该班级的学生名单、成绩统计和预警情况。

### 4.3 课程管理

**核心职责：** 查看和管理所授课程，提供课程推荐功能。

**操作逻辑：**
- 展示教师名下的所有课程列表，包含课程名称、学分、选课人数。
- 支持课程推荐配置，帮助学生选课时获得课程建议。

### 4.4 预警处理与数据分析

**核心职责：** 查看所教班级学生的预警信息，分析教学效果。

**操作逻辑：**
- 预警管理页展示班级预警学生列表，按等级筛选，支持查看详情并添加处理备注。
- 数据分析页提供班级成绩分析、异常检测和学分预测功能，以图表形式展示成绩分布和趋势。

## 五、辅导员端功能模块

辅导员是预警与帮扶的核心执行者，负责预警处理、帮扶计划管理、学分监控和数据分析。

### 5.1 预警管理

**核心职责：** 查看、处理、解除学生的学业预警，是预警闭环的关键执行环节。

**操作逻辑：**
- 列表展示所有预警学生，按预警等级分组，支持批量处理操作。
- 点击单条预警可查看详情：学生基本信息、预警原因、成绩明细、历史预警记录。
- 处理预警时可填写处理意见，更新预警状态。处理完成后预警状态变为"已处理"。
- 当学生成绩改善后，辅导员可手动解除预警，或系统在下次扫描时自动解除满足恢复条件的预警。

**交互关系：**
- 预警生成由 `warning-service` 定时扫描触发，结果推送到 `counselor-service`。
- 辅导员处理后，状态回写到 `academic_warnings` 表，同时通知学生预警处理结果。
- 预警处理流程可衔接帮扶计划模块，辅导员可直接从预警页跳转到帮扶计划创建页。

### 5.2 帮扶计划管理

**核心职责：** 为学业困难学生创建个性化帮扶计划，并全程跟踪执行进度。

**操作逻辑：**
- 创建帮扶计划时，选择目标学生，设定帮扶目标、计划内容、起止日期。
- 帮扶计划创建后，学生端同步可见，学生可查看并更新当前进度。
- 辅导员在计划执行过程中可修改计划内容、更新状态（进行中/暂停/完成）。
- 计划完成后，辅导员进行效果评估；学生可提交帮扶反馈评价。
- 数据分析页提供帮扶计划完成率、平均进度等统计指标。

**交互关系：**
- `counselor-service` 负责帮扶计划的 CRUD 操作，数据存储在 `assistance_plans` 表。
- 帮扶评价数据由学生端提交，经 `student-service` 写入 `assistance_evaluations` 表。
- 帮扶计划和预警管理通过学生 ID 关联，辅导员可从任一模块跳转到另一模块。

### 5.3 学生管理与成绩跟踪

**核心职责：** 查看所管学生的详细信息、成绩走势和学分完成情况。

**操作逻辑：**
- 学生管理页展示所管学生列表，支持搜索和按班级筛选。
- 点击学生可进入详情页，查看该学生的基本信息、每学期成绩明细、预警历程、当前帮扶计划。
- 成绩跟踪页按班级维度展示成绩分布、低分学生名单、成绩趋势图。
- 学分监控页展示学分不足的学生列表，计算达标率和差距。

**交互关系：**
- 学生数据和成绩数据分别从 `student-service` 和 `teacher-service`（`scores` 表）获取。
- 低分学生列表可一键跳转到帮扶计划创建页。

### 5.4 数据分析与通知管理

**核心职责：** 分析预警趋势和帮扶效果，向学生发送通知。

**操作逻辑：**
- 数据分析页提供预警趋势图（按学期统计预警数量变化）、预警分布图（按等级占比）、帮扶计划完成率统计。
- 通知管理页支持向特定班级或全体学生发送通知，可引用预设模板。

## 六、管理员端功能模块

管理员拥有系统最高权限，负责用户管理、基础数据配置、预警规则制定、数据导入导出和权限分配。

### 6.1 用户管理

**核心职责：** 管理系统全部用户，支持增删改查、启用禁用和密码重置。

**操作逻辑：**
- 用户列表展示所有用户，按角色筛选，支持搜索学号/工号/姓名。
- 新增用户时选择角色类型并填写基本信息，系统自动分配权限。
- 禁用用户后，该用户将无法登录系统，其相关数据仍保留。
- 密码重置功能将用户密码恢复为默认密码。

### 6.2 学院、专业、课程管理

**核心职责：** 维护系统的组织架构和教学资源基础数据。

**操作逻辑：**
- 学院管理支持创建、编辑、删除学院，删除时需校验是否有关联数据。
- 专业管理需关联所属学院，支持批量导入。
- 课程管理需关联所属学院和学分信息。

### 6.3 预警规则配置

**核心职责：** 定义预警触发的条件和等级，是所有预警生成的基础。

**操作逻辑：**
- 规则配置页列出所有预警规则，每条规则包含规则名称、触发条件（如"单科不及格""累计挂科≥3门""GPA<2.0"）、预警等级（严重/高/中/低）、启用状态。
- 管理员可新增自定义规则，设定触发条件和对应等级。
- 关闭某条规则后，预警引擎在下一轮扫描中将不再执行该规则。
- 规则生效后，预警引擎每次扫描时按规则列表逐条匹配，命中则生成对应等级的预警。

**交互关系：**
- `admin-service` 管理规则配置，数据存储在 `warning_rules` 表。
- `warning-service` 读取启用状态为"开启"的规则，作为每次扫描的依据。
- 规则变更实时生效，无需重启服务。

### 6.4 数据导入导出与统计分析

**核心职责：** 批量导入学生和成绩数据，导出系统各类报表。

**操作逻辑：**
- 导入功能提供 Excel 模板下载，管理员按要求填写后上传，系统解析并写入对应数据表。
- 导出功能支持按筛选条件导出学生列表、成绩数据、预警记录等。
- 备份恢复功能可一键导出全量数据或从备份文件恢复系统状态。
- 统计分析页展示系统级宏观数据：用户总数、预警总数、处理率、帮扶完成率等，以图表形式呈现。

### 6.5 权限管理与消息管理

**核心职责：** 分配角色权限，向全平台或指定用户发送消息。

**操作逻辑：**
- 权限管理页展示系统所有角色及其拥有的权限列表，管理员可为用户分配角色。
- 消息管理页支持向全体学生/教师/辅导员广播消息，或向特定用户发送定向私信。

## 七、学院管理员端功能模块

学院管理员的管理范围限定在所属学院之内，提供学院级数据概览、班级对比分析和数据导出。

**操作逻辑：**
- 仪表盘展示学院的学生数、班级数、教师数、预警数和平均成绩。
- 点击标签页可分别查看学院内的学生、教师、辅导员、班级、预警、帮扶计划等数据。
- 成绩分析页展示学院各班级的平均分、及格率对比。
- 班级对比页以柱状图形式展示不同班级的成绩差异。
- 数据导出功能支持按学院范围导出学生信息和成绩数据。

## 八、核心引擎模块

### 8.1 学业预警引擎

**核心职责：** 根据管理员配置的预警规则，自动扫描学生成绩数据，生成分级预警。

**运作机制：**
- **定时任务：** 全量扫描任务每天凌晨 2 点执行，扫描所有学生的成绩记录；增量扫描任务每 30 分钟执行一次，仅扫描本周期内新录入或修改的成绩。
- **规则匹配：** 引擎读取所有启用的预警规则，逐条对学生成绩数据进行匹配。匹配逻辑包括：单科成绩不及格、累计挂科数量达到阈值、GPA 低于标准、成绩呈现持续下降趋势等。
- **分级生成：** 根据规则预设等级，生成对应严重程度的预警记录（严重/高/中/低四级），并自动去重——同一学生在同一规则下已存在的预警不会重复生成。

**联动机制：**
- 预警生成后，通过 RabbitMQ 异步通知 `message-service`，同时向学生和对应辅导员推送预警通知（WebSocket + 消息持久化）。
- 辅导员收到预警后，可在管理端处理预警并衔接创建帮扶计划。
- 学生的预警处理状态和成绩改善情况会反馈到预警引擎的下一次扫描中，满足解除条件的预警自动标记为已解除。

### 8.2 AI 智能分析引擎

**核心职责：** 基于阿里云通义千问大模型，提供成绩分析、风险预测、帮扶建议等智能化服务。

**运作机制：**
- 模型选用通义千问 `qwen-plus`，通过 DashScope API 调用。
- 学生和辅导员可在聊天界面输入问题，AI 基于学生的成绩数据、预警历史等上下文进行分析并给出建议。
- 帮扶计划智能生成：AI 根据学生的预警原因和成绩薄弱点，自动生成帮扶计划草案供辅导员参考和修改。

**降级策略：**
- 当 DashScope API 不可用时，系统自动降级到本地规则引擎兜底，确保核心的预警生成功能不受影响。
- 聊天界面显示"AI 暂不可用"提示，不会出现调用超时导致的页面无响应。

### 8.3 消息通知系统

**核心职责：** 实现系统内的实时消息推送和异步通知分发。

**消息类型：**
- 预警生成通知：当预警引擎生成新预警时，自动通知学生和对应辅导员。
- 帮扶分配通知：辅导员创建帮扶计划后，通知目标学生。
- 申诉处理通知：管理员审核申诉后，通知申诉学生结果。
- 广播消息：管理员或辅导员向指定范围用户发送通知。

**推送机制：**
- WebSocket + STOMP 协议实现实时推送，前端在线时即时接收消息。
- 消息同时持久化到数据库，前端离线时可从历史消息中查回。
- 消息状态管理包括未读/已读切换和批量操作。

## 九、端侧覆盖与用户体验

### 9.1 Web 管理端

- 采用 Vue 3 + Vite + Element Plus 技术栈构建。
- 使用侧边栏导航 + 顶部面包屑的经典后台管理布局。
- 支持数据表格、统计图表（ECharts）、表单交互等复杂前端能力。

- 采用 UniApp (Vue 3) 框架开发，一套代码同时支持微信小程序、H5 等平台。
- 底部 TabBar 提供首页、成绩、预警、个人中心四个核心入口。

- 与 Web 端共享同一套后端 API，数据完全同步。

## 十、系统业务闭环

系统通过以下完整闭环实现学业管理的自动化和智能化：

1. **成绩录入：** 教师通过 Web 端或小程序端录入学生成绩，数据同时写入 `score_record` 和 `scores` 两张表。
2. **预警扫描：** 预警引擎定时扫描成绩数据，根据管理员配置的规则自动生成分级预警。
3. **预警生成：** 系统向学生和辅导员推送预警通知（WebSocket 实时推送 + 消息持久化）。
4. **通知推送：** 学生通过小程序/Web 端查看预警详情，了解自身学业风险。
5. **帮扶计划创建：** 辅导员根据预警信息为学生制定个性化帮扶计划。
6. **帮扶执行跟踪：** 学生查看计划内容并更新执行进度，辅导员跟踪全过程。
7. **帮扶评估：** 计划完成后学生提交反馈评价，辅导员评估帮扶效果。
8. **成绩改善：** 帮扶措施见效后，学生成绩提升，预警引擎在后续扫描中自动解除已恢复的预警，形成正向循环。

整个闭环全程由 AI 分析引擎提供智能辅助（成绩分析、风险预测、帮扶建议），数据可视化模块提供实时统计图表，消息通知系统确保各角色之间的信息畅通。
